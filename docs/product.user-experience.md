# Experience Principles

## Core stance

Hakumi should feel calm, capable, and continuous.

The product should reduce fragmentation, not add to it. Every screen should reinforce the feeling that the user is working inside one coherent workspace with one memory.

## Principles

### Notes first

Notes are the center of gravity.

Chat, search, tags, voice capture, and future systems should make notes more useful. They should not compete with notes for product identity.

### Capture before structure

The fastest path into the product should be low-friction capture.

People should be able to save something incomplete without deciding too much. Structure should emerge later through editing, linking, context, and use.

### One feed, one workspace

The product should present one unified place where work accumulates.

Notes and chats should feel like different shapes of the same underlying activity, not like separate products with separate homes.

### Chat is a thinking tool

Chat should help users reason with their own context.

It should not feel like a generic assistant bolted onto a notes app. It should feel like a way to ask the workspace what it knows, continue an idea, or turn context into the next step.

### Calm over chrome

Visual weight should come from hierarchy, spacing, and restraint.

The interface should avoid noisy containers, decorative badges, and multiple competing signals. The product should feel stable and native rather than flashy.

### Recognition over interpretation

Lists and navigation should optimize for fast recognition.

Users should be able to scan titles, time, and type quickly. Extra metadata, snippets, or labels should appear only when they materially improve understanding.

### Context should stay nearby

Relevant notes, attached context, and conversational history should stay close to the active task.

The product should minimize the feeling of jumping between disconnected surfaces to recover what the user was already doing.

### The interface should recede when focus matters

When a user is editing a note or working through a conversation, the product should let the primary object take over. Supporting UI should stay available without competing for attention.

### Motion should clarify, not decorate

Transitions should help users understand continuity, focus changes, and system response.

Motion should be restrained and purposeful. It should make the workspace feel alive without making it feel busy.

## Canonical language

Use these terms consistently in product docs:

- `feed` for the unified chronology
- `chat` for the conversational thinking surface
- `workspace` for the product as a whole
- `notes-first personal workspace` for the core product framing

Avoid using `focus` and `stream` as primary product terms in documentation. They can remain internal implementation names if needed, but the product story should stay simpler.
