# Pre-Build Cleanup Tasks

**Scope:** Hominem monorepo, with focused attention on `apps/omiro`.

**Goal:** Establish a clean, understood, reproducible baseline before starting new product work.

**Rule:** Prefer deletion and one-line simplifications. Do not change behavior unless the task explicitly requires it. Omiro remains Apple-only.

## Phase 1: Repository Hygiene

- [ ] Audit the working tree for uncommitted changes, generated files, and accidental local artifacts.
- [ ] Identify tracked build output such as `dist`, `build`, `.expo`, Pods, derived data, and simulator artifacts.
- [ ] Confirm generated native files are either intentionally committed or regenerated by the documented Expo workflow.
- [ ] Review stale branches, abandoned worktrees, and old feature directories.
- [ ] Find duplicate configuration files, obsolete scripts, and documentation that describes removed systems.
- [ ] Remove only artifacts confirmed to be generated and ignored by the intended workflow.
- [ ] Document any intentionally tracked native files and the command that regenerates them.

## Phase 2: Dependency And Dead-Code Audit

- [ ] Run Knip across the entire monorepo, not only `apps/omiro`.
- [ ] Review unused files, exports, dependencies, and duplicate entry points package by package.
- [ ] Remove unused direct dependencies only after checking native config plugins and transitive peer requirements.
- [ ] Review workspace packages that contain only re-exports or theoretical shared code.
- [ ] Identify duplicate helpers, wrappers, barrels, and path aliases that add no ownership or behavior.
- [ ] Decide whether the duplicate relative-date path can be collapsed:
  - `apps/omiro/services/date/format-relative-age.ts`
  - `apps/omiro/services/date/date/format-relative-age.ts`
- [ ] Check for circular and soft-circular imports between workspace packages.
- [ ] Re-run dependency installation with a frozen lockfile after every dependency cleanup.

## Phase 3: Omiro Expo And Native Audit

- [ ] Compare `apps/omiro/app.config.ts` with the actual iOS targets and runtime imports.
- [ ] Review remaining config-only packages and decide whether each is required:
  - `expo-asset`
  - `expo-notifications`
- [ ] Confirm every Expo config plugin has a current product or native-build purpose.
- [ ] Audit iOS permissions, entitlements, URL schemes, deployment targets, and Info.plist entries for unused declarations.
- [ ] Audit `@bacons/apple-targets` configuration against the Control Center target.
- [ ] Confirm `OmiroIntents` and `VoiceTranscriber` are both used at runtime and remain Apple-only.
- [ ] Compare `ios/Podfile.lock` and the generated Xcode project against the current package/config state.
- [ ] Verify the Metro package-export fallback remains intact for shared ESM packages.
- [ ] Verify EAS configuration does not pin `pnpm` in `apps/omiro/eas.json`.
- [ ] Run the production embed command:

  ```bash
  pnpm --filter @hominem/omiro exec expo export:embed --eager --platform ios --dev false
  ```

- [ ] Confirm the generated iOS bundle contains only intended native modules and assets.
- [ ] Run the critical Maestro flows on the configured iPhone simulator, using `id` selectors for modal controls.

## Phase 4: Architecture And Data Flow Audit

- [ ] Trace authentication from Better Auth through signed cookies, native storage, boot, session refresh, and sign-out.
- [ ] Confirm Better Auth remains the sole authentication authority.
- [ ] Remove or isolate legacy auth/session compatibility code that is no longer on a live path.
- [ ] Audit API and RPC boundaries for duplicated request, response, and error contracts.
- [ ] Check that mobile query hooks own cache keys and invalidation consistently.
- [ ] Identify direct API calls that bypass the typed RPC boundary.
- [ ] Review shared packages for business logic that is duplicated between mobile, web, and API.
- [ ] Audit runtime schemas at external boundaries and remove decorative duplicate schemas from internal code.
- [ ] Check database models, repositories, services, and RPC outputs for shape drift.
- [ ] Check migration history against the current schema and identify migrations that are missing, duplicated, or undocumented.

## Phase 5: Testing And CI Audit

- [ ] Map the critical user flows and ensure each has an appropriate unit, integration, or Maestro test.
- [ ] Remove duplicate or obsolete tests and test fixtures.
- [ ] Identify flaky tests and record the failure mode before changing them.
- [ ] Verify tests do not depend on production credentials, real user data, or shared mutable state.
- [ ] Review every GitHub Actions workflow for stale paths, redundant jobs, and unsupported action versions.
- [ ] Confirm CodeQL runs on the intended JavaScript/TypeScript scope and that alert results are re-baselined after the cleanup.
- [ ] Add or verify dependency, secret, and lockfile scanning in CI.
- [ ] Verify a clean checkout can install, typecheck, build, and test without local-only files.

## Phase 6: Security And Operations Audit

- [ ] Audit environment variables for unsafe defaults, undocumented secrets, and client exposure.
- [ ] Verify `AUTH_TEST_OTP_ENABLED` is explicitly `false` in production and defaults to test-only behavior.
- [ ] Confirm OTP, token, cookie, and credential values never reach logs or analytics.
- [ ] Review session cookie settings, native storage, expiration, and sign-out behavior.
- [ ] Audit file uploads, URL validation, local file access, and temporary-file cleanup.
- [ ] Review user-controlled values passed to logs, shell commands, URLs, regular expressions, and filesystem operations.
- [ ] Review native permissions and privacy declarations for least privilege.
- [ ] Check dependency vulnerabilities and native package advisories before feature work begins.
- [ ] Document production incident checks without retrieving session tokens or user records.

## Exit Criteria

- [ ] No unexplained tracked generated artifacts remain.
- [ ] No high-confidence unused dependencies or files remain without an owner-approved reason.
- [ ] Omiro config, native targets, and package dependencies agree.
- [ ] Authentication, API, database, and query boundaries are documented and consistent.
- [ ] Critical mobile flows pass in Maestro on the configured iOS simulator.
- [ ] Full formatting, lint, typecheck, build, and test validation passes.
- [ ] CodeQL and dependency scans have no unresolved actionable findings.
- [ ] A clean checkout reproduces the same validation results.

