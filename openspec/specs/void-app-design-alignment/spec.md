# void-app-design-alignment Specification

## Purpose
Align the mobile, desktop, and notes app surfaces to a single canonical design language so shared flows and components render consistently, accessibly, and without platform-specific visual drift.

## Requirements
### Requirement: Single Light Mode Only
App surfaces SHALL use a single light color scheme. Dark mode utilities and `prefers-color-scheme: dark` are forbidden.

#### Scenario: No dark mode utilities in app code
- **WHEN** auditing app component class names or CSS
- **THEN** no `dark:` Tailwind prefix or `:dark` media queries are present

### Requirement: App color values SHALL use design tokens
All color values in app code SHALL come from the approved design token system (`@hominem/ui/tokens/colors.ts` or Tailwind design-token config).

#### Scenario: No raw hex or rgba values
- **WHEN** app code defines or uses a color (in JSX `style` attr, CSS, or Tailwind `text-[]` syntax)
- **THEN** the color is imported from or generated by the design token system, never hardcoded

### Requirement: App typography SHALL use the canonical scale
All app text SHALL use font sizes from the approved typography scale (`@hominem/ui/tokens/typography.ts`) and follow rules: body/prose text ≥17px, no fixed-height containers that break sizing overrides.

#### Scenario: Typography uses canonical scale
- **WHEN** a component renders text (headings, labels, body)
- **THEN** the text uses a canonical typography class or size, and is not constrained by a rigid container height

### Requirement: App motion and animation SHALL use canonical primitives only
All component motion and transitions in apps SHALL use canonical animation primitives (web: CSS classes from `animations.css`; mobile: constants and hooks from `motion.ts`). Custom `@keyframes`, ad-hoc `transition`, or `animation` properties are forbidden.

#### Scenario: Enter/exit animation uses canonical classes
- **WHEN** a component appears or disappears
- **THEN** it uses `.void-anim-enter` or `.void-anim-exit` (or directional variants) or mobile hooks such as `useVoidEnter()` or `useVoidExit()`

#### Scenario: Hover interaction does not use transforms
- **WHEN** a user hovers over an interactive element
- **THEN** the element does not scale, rotate, or translate as a decorative effect

### Requirement: App motion SHALL respect prefers-reduced-motion
All animation in apps SHALL respect the user's motion preference. Components using canonical primitives inherit this compliance; custom animation must be explicitly disabled when `prefers-reduced-motion` is active.

#### Scenario: Motion disabled for users with reduced-motion preference
- **WHEN** a user has enabled `prefers-reduced-motion` system setting
- **THEN** animations are disabled or simplified

### Requirement: App visual depth treatments SHALL be minimal and purposeful
Apps SHALL avoid decorative depth: no rounded corners used only for ornamentation, no shadows except where explicitly required by the design system, and no blur effects.

#### Scenario: No decorative rounded corners
- **WHEN** auditing app component styling
- **THEN** rounded corners are used only for functional visual separation, not as default decoration

#### Scenario: No decorative shadows
- **WHEN** auditing app component styling
- **THEN** shadows are used only when required by the design system for visual hierarchy, not for ornamental depth

### Requirement: App focus states SHALL meet WCAG 2.2 AA accessibility
All interactive elements in apps SHALL have visible `:focus-visible` states with sufficient contrast for WCAG 2.2 AA compliance.

#### Scenario: Focus indicator is visible and accessible
- **WHEN** a user navigates to an interactive element with the keyboard
- **THEN** the focus ring or indicator is clearly visible with sufficient contrast

### Requirement: App text and interactive element contrast SHALL meet WCAG 2.2 AA
All text in apps SHALL have ≥4.5:1 contrast against its background, and interactive element boundaries SHALL have ≥3:1 contrast.

#### Scenario: Text is readable
- **WHEN** a user reads text on any app surface
- **THEN** the text meets WCAG 2.2 AA contrast requirements

### Requirement: App icons and decorative elements SHALL communicate information or be removed
Icons in apps SHALL only be used when they communicate information. Decorative emojis, symbols, or visual flourishes that do not add semantic meaning are forbidden.

#### Scenario: Icon usage is semantic
- **WHEN** an icon or emoji appears in the app UI
- **THEN** it communicates a piece of information or is removed

### Requirement: App design SHALL be 1-1 aligned across mobile, desktop, and notes platforms
Any UI component or user-facing flow that appears on multiple platforms SHALL render and interact identically unless an explicit platform-specific exception is documented.

#### Scenario: Component renders identically across platforms
- **WHEN** viewing the same component on mobile, desktop, and notes apps
- **THEN** the visual hierarchy, spacing, typography, color, focus state, and motion are identical

#### Scenario: User flow matches across platforms
- **WHEN** a user completes the same workflow on mobile, desktop, and notes
- **THEN** the visual feedback, form validation, error messaging, and success confirmation are identical

#### Scenario: Platform-specific features are documented
- **WHEN** a component or interaction differs between platforms
- **THEN** the exception is documented with the platform-specific reason
